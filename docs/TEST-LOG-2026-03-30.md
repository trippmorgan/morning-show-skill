## Test Results — 10:01 AM EDT

### Test A — Hour 10 (Full Hour, Manual Markers)
- **Started playing?** ❌ NO — instant skip
- **Root cause:** `Audio.Filename = '90001.mp3'` but no file by that name on disk. Actual file was `MORNING-SHOW-H1.mp3`. The SQL registration script failed earlier (invalid column "Chain"), so the Filename was never updated from the previous session's value.
- **PlayoutONE behavior:** Marked entry as Played=True, Done=True instantly (file not found → skip)
- **Station impact:** None — AutoFill loaded normal music immediately
- **Fix applied:** Copied `MORNING-SHOW-H1.mp3` → `90001.mp3` for future tests

### Test B — Hour 11 (10-min Segment, Manual Markers)
- **Started playing?** ❌ OVERWRITTEN — never got a chance
- **Root cause:** Music1 ran at ~07:12 and regenerated DPLs for ALL 24 hours. The Music1 DPL for hour 11 was imported AFTER our test DPL, overwriting our custom entry.
- **Hour 11 playlist shows:** UID 16298 "It's Too Late" (Music1 entry) instead of UID 99901 (our test segment)
- **99901.mp3 file:** Exists on disk ✅
- **99901 Audio table:** TrimOut=599980, Extro=594980 ✅ (manually fixed)

### Key Findings

1. **File naming is critical:** `Audio.Filename` must match the actual file on disk. PlayoutONE instant-skips if file not found (no error, just marks Played/Done and moves on).

2. **Music1 DPLs overwrite custom DPLs:** If Music1 generates a DPL for the same hour as a custom show, the LAST one imported wins. Music1 ran after our test DPLs were deployed, overwriting them.

3. **AutoImporter does NOT analyze audio:** Confirmed — TrimOut and Extro must always be set manually in the Audio table regardless of file size.

4. **Import order matters:** The DPL import folder processes files... but if Music1 drops 24 fresh DPLs, they overwrite everything.

### Publish Strategy Implications

For the morning show to survive Music1:
- **Option A:** Schedule Music1 to run BEFORE the show publish. Then our DPLs overwrite Music1's.
- **Option B:** Delete our target hour DPLs from the import folder after Music1 runs (race condition risk).
- **Option C:** Don't use DPL import for show hours — use SQL to modify the Music1-created entries AFTER import (but this is what crashed us before).
- **Option D:** Exclude show hours from Music1 entirely (modify Music1 config if possible).

**Recommended:** Option A — run Music1 first, then publish show DPLs second. Our DPLs overwrite Music1's entries for those hours.

### Critical Discovery: AutoImporter Skips Existing Hours

When DPLs for hours 12 and 13 were dropped into the import folder, AutoImporter processed them (moved to Imported/) but did NOT create Playlists entries — because entries for those hours already existed from Music1.

**AutoImporter rule: First import wins. It will NOT overwrite existing entries for an hour.**

### Correct Publish Sequence (Tested & Working)

1. DELETE existing Playlists entries: `DELETE FROM Playlists WHERE GIndex LIKE 'YYYYMMDD{HH}%'`
2. Remove old DPL from Imported folder: `Remove-Item '...Imported\YYYYMMDDHH.dpl'`
3. Drop custom DPL into import folder
4. AutoImporter picks it up and creates new entries ✅
